Mix is easy

A simple 'hello world' can be printed in two ways:
  • Using a module
  • Directly printing it from the entry initialisation point

    • Indentation in Mixlang is optional, but if you do choose to indent, it must be consistent.

    By that, I mean the arrows must be indented correctly, and should only be at the start of the file, following previous indentation (if any).

    To better explain this, look at the example below:

    
[entry.Init] _: like the main function in C/C++. it is also case sensitive

-->
out ;hello, world; _: printing hello world to the console

$exit::0 _: return 0
<--

[entry.Stop] _: terminating the script.

    This example is correct. However, the example below isn't:

    
[entry.Init] _: like the main function in C/C++. it is also case sensitive

-->
out ;hello, world; _: printing hello world to the console

  $exit::0 _: this is indented inconsistently, and will result in an error
<--

[entry.Stop] _: terminating the script.

    +Cursors & -Cursors

    They tell the Mux JIT compiler when to start construction of code and blocks of code. The role of Plusors and Minusors is very useful. Take this example:

    
[entry.Init]

-->
call ;main; _: calling the main module (not to be confused with the script entry initialisation point)
call ;helloworld2; _: the code is executed procedurally
<--

.module ;main;-> _: a '->' shows the starting point of whatever function, script etc. not to be confused with +C or -C
-->
out ;hello, world;

;if 20 ?: 5->
-->
    out ;20 equal to 5;
<--

;else
-->
    out ;20 unequal to 5;
<--

<--

.module ;helloworld2;->
-->
out ;hello world from second module;
<--

[entry.Stop] _: terminating the script.

    What is Mix Language

    The Mix programming language is a programming language I programmed in Python. Its interpreter is written in Python, but compiled as standalone executable so users don't need Python on their system.

    What Makes it Special

    • No need to import libraries (90% of them are preimported)
    • Automatic, QoL functions, such as `$erase` and `$wait::{time_in_seconds}`

    Exit Codes

    • Exit code -1: Usually a syntax error or an error in the Mix file
    • Exit code -2: Either your MIX file is corrupt or doesn't exist
    • Exit code -3: The MIX file contains an unopened block, no block, or a wrongly-coded block
    • Exit code -4: The MIX file contains an unterminated block if the block is already opened
    • Exit code -5: The MIX file contains an unterminated literal expression or wrong usage of parentheses
    • Exit code -6: There was an issue running the MIX file due to insufficient permissions or as such
    • Exit code -7: System could not resolve error
    • Exit code -8: Either you opened a module with an unterminated literal expression, or you forgot to open a module
    • Exit code -9 *(*Old)*: You forgot to initialize the script
    • Exit code -10: You forgot to terminate the script
    • Exit code -11: There was an unexpected issue in the M4M-Interpreter (Mux for Mix)
    • Exit code -12: There was an unexpected issue encountered whilst running the script, or inside the script, or you forcefully exited the process/app

    Basic Mix

    Chapter 1: Hello, World!

    
[entry.Init]
-->
out ;Hello, world!;
<--
[entry.Stop]

    Chapter 2: Cursors

    Let's take a look at cursors, and their role in MIX:

    • Fundamentally, a cursor allows you to define when to start executing code and when to stop
    • Additionally, a cursor is also the main defining factor of a 'code-block', much like curly braces in other languages
    • There are two types of cursors in MIX:
      • Plusor (plus-cursor): This allows you to define the start of a code-block, and looks like '-->'
      • Minusor (minus-cursor): This allows you to define the end of a code-block, or reset the code-block, and looks like '<--'

    What do I mean by resetting code-blocks?

    You must additionally use '<--' after ending a tagged-code-block. A tagged code-block is one in which you run additional scripts.

    Chapter 3: Entry points

    An entry point refers to the earliest 'point of entry' in your Mix app. There cannot be two entry points, or more. An entry-point is defined at the top of the Mix app.

    An entry point serves as instructions to the interpreter on how to, procedurally, execute instructions or modules. You cannot directly call a hello world function.

    If you wish to skip the step of making an extra module, you can do it all inside the entry point pool itself, but that is not recommended because the code becomes literally unreadable after a while.

    An entry point is defined by:

    
[entry.Init]

    This will initialize your script.

    An exit point is defined by:

    
[entry.Stop]

    This will terminate the script
